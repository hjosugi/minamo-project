## 1. 目標

2D/3Dアバターは、安価なハードウェアで迅速に読み込む必要があります。

## 2. 3Dアセット

推奨GLBパイプライン:

1. アセットを検査する
2. 未使用のノードを削除する
3. アニメーションを再サンプリングする
4. 安全な場合はテクスチャをKTX2に圧縮する
5. モデルごとにジオメトリにmeshoptまたはDracoを選択する
6. スケルトンとブレンドシェイプの名前を保持する
7. ビジュアル回帰を実行する

リグ付きアバターを盲目的に最適化しないでください。一部の最適化ステップはノードの階層を変更したり、ブレンドシェイプのマッピングを壊したりする可能性があります。

### GLB検査チェックリスト

アバターを変更する前に記録する:

- ファイルサイズ、テクスチャ数、テクスチャの寸法、マテリアル数
- ノード数、メッシュプリミティブ数、頂点数、モーフターゲット数
- ヒューマノイドボーンの名前とVRM拡張バージョン
- 表情/ブレンドシェイプの名前
- スプリングボーンコライダーとジョイントの数
- アニメーションクリップの名前と持続時間（存在する場合）

ツールが`VRMC_vrm`、`VRM`、モーフターゲット、ヒューマノイドノード、またはスプリングボーン拡張を削除する場合、検査に失敗します。

依存関係なしで最初のパスを行うために、組み込みのインスペクターを使用してください:

```bash
pnpm inspect:glb -- avatar.glb
pnpm inspect:glb -- avatar.glb --json
pnpm inspect:glb -- avatar.glb --avatar
```

`--avatar`は、ファイルがVRMメタデータ、ヒューマノイドボーンマッピング、モーフターゲット、または表情名を欠いている場合に非ゼロで終了します。これは、glTF Transformまたはアーティストツールがファイルを書き換える前のプレフライトガードとして意図されています。

### glTF Transformガイド

GLBファイルを予測可能に検査および書き換えることができるため、最初の最適化ツールとしてglTF Transformを使用します:

```bash
gltf-transform inspect avatar.glb
gltf-transform dedup avatar.glb avatar.dedup.glb
gltf-transform prune avatar.dedup.glb avatar.pruned.glb
```

プルーニングされたファイルがビューワーでまだ読み込まれる場合にのみ、テクスチャとメッシュの圧縮を実行します。

### Meshopt vs Dracoの決定

| 選択肢 | 使用する場合 | 避ける場合 |
|---|---|---|
| meshopt | リアルタイムWebビューワー、アニメーションアバター、良好なデコード速度 | アセットパイプラインが拡張の順序を保持できない場合 |
| Draco | 静的メッシュ、最大ジオメトリ圧縮 | モーフが多いアバターや遅いモバイルデコードパス |

アバターにはデフォルトでmeshoptを使用します。ビジュアル回帰がブレンドシェイプとスプリングボーンが生き残ることを確認した後にのみDracoを使用します。

推奨される決定記録:

- 元のファイルと最適化されたファイルのバイトサイズ
- 低スペックデバイスでの最初のフレームのビューワー読み込み時間
- Chromeパフォーマンスプロファイルでのデコード時間
- インスペクターのサマリーにすべての表情がまだ存在するかどうか
- スプリングボーンのジョイントとコライダーが元の数と一致するかどうか

### KTX2テクスチャガイド

KTX2/BasisUは、大きなアルベドおよび法線テクスチャに適しています。以下のために非圧縮PNG/WebPのフォールバックを保持します:

- 小さなUIテクスチャ
- シャープなアルファマスクを持つテクスチャ
- 作成ツールがマテリアル参照を保持できないリグ

GPUメモリと最初のフレームの読み込み時間を前後で記録します。

テクスチャ受け入れチェックリスト:

- アルベドマップはブロックアーティファクトなしに肌のグラデーションを保持する
- 法線マップは顔や衣服のきらめきを引き起こさない
- アルファカットアウトテクスチャはOBSの透明モードでクリーンなエッジを保持する
- マテリアル参照は意図した画像スロットを指し続ける
- モバイルブラウザは選択したトランスコードターゲットをフォールバックの遅延なしで読み込む

## 3. 2Dアセット

- テクスチャアトラスのマニフェストを決定論的に保つ: 安定したレイヤー名、スロット、深さ、元のサイズ、パックされた長方形、および変換の原点。
- 古いGPU用に2の累乗のアトラスオプションを提供しますが、デバッグ用にソースPNG/PSDレイヤーをエクスポート可能に保ちます。
- アルファエッジを確認した後にのみ圧縮テクスチャのバリアントを生成します。
- 現在のプリセットでアクティブでない表現パックは遅延読み込みします。
- アトラスの再構築間でリグパラメータ名を安定させます。

レイヤー付きPNGおよびPSDモードの場合、同じ変換の原点を共有する静的レイヤーのみをアトラスします。アトラスパッキングがデバッグを難しくする場合は、口/目のスワップレイヤーを別に保ちます。

## 4. モーションフレーム

KGM1圧縮ステージ:

- 正規化された値を量子化する
- キーフレームからデルタエンコードする
- 低信頼度の未使用信号を省略する
- ドラムヒット用のイベント専用パケット
- 定期的な信頼できるキーフレーム

モーションデルタ量子化ポリシー:

- 顔の表情ウェイト: 最後のキーフレームからの8ビット正規化デルタ
- 頭の回転: 最短経路のクォータニオンデルタ、バイナリの場合はコンポーネントごとに12-16ビット
- 手のカール/スプレッド: リグクランプ後の8ビット正規化値
- ドラムヒット: イベントパケットのみ; アイドルドラム状態をストリーミングしない
- 再接続、モデル変更、または2秒の連続デルタの後にキーフレームを強制する

量子化受け入れゲート:

- ニュートラルな顔が目に見える口やまばたきのドリフトなしで往復する
- 視線方向の誤差がデルタデコード後に3度未満に留まる
- 手のカールの誤差がクランプ後にリグで見えるステップ未満に留まる
- 再接続キーフレームが1フレームで完全な状態を復元する
- 古いデルタは新しいキーフレームの後に到着した場合は破棄される

## 5. 品質のトレードオフ

顔と口の信号は通常、詳細なランドマークよりも高い優先順位を受けるべきです。リモートストリーミングの場合、アバター準備パラメータを最初に送信し、生のランドマークを2番目に送信します。

## 6. ビジュアル回帰チェックリスト

すべての最適化されたアバターは元のものと照らし合わせてチェックする必要があります:

- ニュートラルポーズ
- 左/右まばたき
- 口を開ける、笑顔、口をすぼめる
- 左/右/上/下を見る
- 頭を回した後のスプリングボーンの動き
- 透明なOBS背景
- 1つの低スペックデバイステスト

元のアバターと最適化されたアバターを同じビューワーURLからキャプチャします。各ポーズについて、フルフレーム画像と切り抜かれた顔画像を比較します。ピクセル差分ツールを使用する場合は、スプリングモーションが顔やマテリアルの回帰を隠さないように、背景と髪の先端をマスクします。

## 7. アセットライセンスチェックリスト

バンドルされたアセットまたはサンプルアセットごとに記録する:

- ソースURLまたは著者提供のパッケージ
- ライセンス名とバージョン
- 再配布が許可されているかどうか
- 修正/圧縮が許可されているかどうか
- 表示が必要かどうか
- モデル出力またはスクリーンショットがドキュメントで使用できるかどうか

ライセンス記録をアセットの横または外部サンプルファイルのリリースノートに保存します。圧縮、リターゲティング、およびスクリーンショットは、多くのクリエイターマーケットプレイスライセンスにおいて修正または派生使用と見なされます; 再配布と修正の両方が許可されるまで最適化されたサンプルを公開しないでください。

## 8. kagami-pack CLI (KGM-041)

`kagami-pack` (`scripts/kagami-pack.mjs`) は、ホストされたアバターパイプラインのプランナーおよびレポーターです。GLB/VRMを検査し、保守的なステージ順序を計画し、正確なエンコーダコマンドを出力し、前後のテーブルをレンダリングします。

```bash
pnpm pack:avatar -- avatar.glb            # meshopt + KTX2プラン
pnpm pack:avatar -- avatar.glb --draco    # meshoptの代わりにDracoジオメトリ
```

プランナー（`planAvatarPack`）はリグに重要なカウントを記録し、ロスのあるステップの前に警告します; オペレーターは出力された`gltf-transform` / `gltfpack`コマンドを実行し、その後`pnpm inspect:glb -- <out> --avatar`で再検査します。`formatSizeTable`を使用して、前後のサイズとGPUメモリを報告します:

| メトリック | 前 | 後 | 差 |
| --- | --- | --- | --- |
| ファイルサイズ | 8.00 MB | 2.40 MB | -70.0% |
| GPUメモリ | 120 MB | 40 MB | -66.7% |

上記の数値は参考値です（典型的なVRMは60-80%縮小します）；ビジュアル回帰パスの後にアセットごとに測定値を記入してください
([visual-regression-checklist.md](visual-regression-checklist.md)).
