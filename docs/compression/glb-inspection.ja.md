# GLB検査スクリプト設計

ステータス: イシュー #156 の設計が実装されました。関連: #41。

検査はアバター圧縮パイプラインの最初のステージです
([avatar-compression.md](avatar-compression.md))。アバターが検査され、そのリグに重要な構造が記録されるまで、何も書き換えられません。

## ステップ

1. 依存関係のないインスペクターをプレフライトとして実行します:

   ```bash
   pnpm inspect:glb -- avatar.glb
   pnpm inspect:glb -- avatar.glb --json
   pnpm inspect:glb -- avatar.glb --avatar
   ```

2. 最適化の前に、すべてのアバターのベースラインサマリーを記録します:
   ファイルサイズ、テクスチャ数と寸法、マテリアル数、ノード数、メッシュ
   プリミティブ数、頂点数、モーフターゲット数、ヒューマノイドボーン名、VRM
   拡張バージョン、表情/ブレンドシェイプ名、スプリングボーンコライダー/ジョイント
   数、アニメーションクリップ名と持続時間。
3. ベースラインJSONをソースアセットの隣に保存し、後でそれに対して
   差分を取れるようにします。
4. 各オプティマイザーステージの後に再検査を行い、サマリーフィールドを
   項目ごとに比較します。静かに変更されてはならないカウントは、モーフターゲット、
   ヒューマノイドノード、スプリングボーンジョイント、および表情名です。

パーサーとサマリーのヘルパーは
[`scripts/inspect-glb.mjs`](../../scripts/inspect-glb.mjs) からエクスポートされており
(`parseGlb`, `summarizeGltf`, `formatInspection`)、同じロジックがCLI、テスト、
およびチェックリスト評価者を支えています。

## リグ破損のリスク

- `--avatar` モードは、ファイルにVRMメタデータ、ヒューマノイドボーンマッピング、
  モーフターゲット、または表情名が欠けている場合、非ゼロで終了します。
  非ゼロの終了を警告ではなくハードストップとして扱います。
- `VRMC_vrm`、`VRM`、モーフターゲット、ヒューマノイドノード、またはスプリング
  ボーン拡張を削除するツールは、メッシュがまだレンダリングされていても
  検査に失敗します。
- 名前が変更されたヒューマノイドボーンや表情は、下流のリターゲティングや
  マッピングエディタを破損させます; カウントが一致していても名前の変更は
  破損としてカウントされます。

## テスト方法

- `pnpm test` は合成GLBに対して `parseGlb`/`summarizeGltf` を実行し、
  チェックリストで使用されるサマリーフィールドを検証します。
- サンプルアセットチェックリスト (`shared/compression-checklist.js`,
  `evaluateAssetChecklist`) は記録された検査サマリーをリグ保存ゲートを通過させ、
  リグに重要なカウントがゼロに落ちると失敗します。
- 手動: 元のアセットと検査されたアセットをビューワーで開き、
  報告された表情とボーン名がアバターを駆動し続けていることを確認します。
