# 研究: IMUスティックのオプション統合

ステータス: イシュー #185 のための研究設計。MVPの範囲外; 設計文書として保持。
関連: [../tracking/drum-performance-tracking.md](../tracking/drum-performance-tracking.md),
#183。

## 目標

ドラムスティックにオプションのIMU（加速度計/ジャイロ）を追加して、ヒットタイミングと速度を改善することを評価し、ウェブカメラファーストで追加ハードウェアに依存しない設計に適合するかどうかを決定する。

## 受け入れ基準

- [x] 目標が明確である。
- [x] 受け入れ基準が明確である。
- [x] 既存の設計と矛盾しない: IMUは厳密にオプションであり、ウェブカメラ + オーディオパスはデフォルトであり、ハードウェアに依存しない。
- [x] ドキュメントレビュー ([PR #220 レビュー記録](../verification/pr-220/research-review/185.md))。
- [x] 手動検証: サポートされているアクセサリーやプロトタイプが存在しないため、この研究専用の決定には `N/A`; フォローアップ #240 がハードウェアプロトタイプのゲートを所有。

## 調査結果

- IMUはほぼゼロ遅延のインパクト検出とピーク加速度からのクリーンな速度を提供し、カメラパスはそれを近似することしかできない。
- コストは実際のハードウェア、スティックごとのペアリング、バッテリー充電/テレメトリ、ファームウェア/バージョンサポート、再接続処理、およびBLEまたはシリアルトランスポートに関わる。
- Web Bluetoothは一部のブラウザ/プラットフォームの組み合わせでのみ利用可能であり、安全なコンテキストとユーザーのジェスチャーを必要とし、Safari/WebKitでは実装されていない。したがって、ブラウザ専用のアクセサリーは普遍的なパスにはなり得ない。 [Chromeの実装ガイダンス](https://developer.chrome.com/docs/capabilities/bluetooth) と [WebKitの現在の立場](https://webkit.org/tracking-prevention/)を参照。
- IMUデータは同じヒット融合/クールダウンステージへの別の信号であり、`DrumHitEvent` スキーマを変更することはなく、その信頼性とタイミングソースのみを変更する。

## 決定

IMUを**MVPから除外**し、オプション機能の背後に置く:

- IMUを視覚と音声と融合した補助的な発生/速度信号として扱い、決して必須とはしない。
- プロトタイプが作成された場合、Web Bluetoothを機能検出し、明示的なサポートされていない診断を表示する。WebKitおよびサポートされていないデスクトップブラウザ用にネイティブ/Tauri BLEブリッジを評価する; オプションのアクセサリーが存在しない場合に視覚/音声が劣化することを暗示しない。
- ペアリングはユーザーが開始し、選択されたデバイス/サービスにスコープされ、公開ログに識別子を保持せずに切断から回復する必要がある。
- バッテリーレベル、パケットレート、ドロップされたサンプル、クロックドリフト、および再接続がクールダウン/ダブルヒット動作に与える影響を記録する。
- 既存の共有クロックオフセット推定を再利用し、IMUのタイムスタンプがカメラとオーディオに合わせて整列するようにし、マルチカメラ融合のために提案されたのと同じアプローチ (#183)。

この研究パスにはプロトタイプコードは存在しない。特定のIMUスティック、GATTプロファイル、ファームウェア/ライセンスポリシー、およびサポートされているブラウザ/ネイティブブリッジマトリックスがフォローアップ #240 で提案された場合にのみ再検討する。
