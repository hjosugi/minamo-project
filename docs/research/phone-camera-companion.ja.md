# 研究: 電話カメラコンパニオンアプリ

ステータス: イシュー #184 の研究設計。関連: #51 (電話をトラッカーとして使用するモード),
[../product/phone-tracker.md](../product/phone-tracker.md), #183。

## 目標

電話が Minamo のコンパニオンキャプチャデバイスとして機能すべきか評価し、
ネイティブコンパニオンアプリと既存のブラウザトラッカーのどちらを選択するか決定する。

## 受け入れ基準

- [x] 目標が明確である。
- [x] 受け入れ基準が明確である。
- [x] 既存のデザインと矛盾しない: 生のビデオは電話から出ない;
  KGM モーションフレームのみが公開され、デスクトップトラッカーと全く同じである。
- [x] ドキュメントレビュー ([PR #220 レビュー記録](../verification/pr-220/research-review/184.md))。
- [ ] 手動検証は #226 (QR UI)、#227 (セキュアトランスポート交渉)、および #228 (リアルiPhoneタイミング) によって `BLOCKED` されている; これらの結果はこの研究決定によって主張されていない。

## 発見

- 電話はしばしばノートパソコンよりも優れたカメラとデバイス上の ML を持っているため、顔や手の品質のためにキャプチャデバイスとしての電話は魅力的である。
- ブラウザトラッカーは意図されたモバイル実装パスであるが、このパスは検証済みの電話プロトタイプを主張していない。QR ペアリング、セキュア HTTPS-to-WSS/WT トランスポート、小画面の動作、リアルデバイスタイミングは #51 の下で #226–#228 に追跡されている。
- ネイティブアプリは App Store/Play Store 配布、ネイティブカメラ制御、バックグラウンドの安定性を追加するが、大きなメンテナンスコストと二重のコードベースが必要となり、「シングルブラウザファーストトラッカー」という方向性と矛盾する。

## 決定

**ブラウザコンパニオン** (プログレッシブウェブアプリ) を優先し、ネイティブアプリは使用しない:

- 既存のトラッカーを再利用し、QR ペアリング (#51) とモバイルレイアウトパスを追加する。
- プライバシー契約を同一に保つ: デバイス上の推論、既存の WS/WebTransport トランスポートを介したモーションのみの公開。
- 電話ページから HTTPS とセキュア WSS/WT を要求する; HTTPS ページからのプレーン WS はサポートされているフォールバックではない。
- 具体的な機能 (真のバックグラウンドキャプチャ、ブラウザがアクセスできないカメラ API) が厳しい要件となった場合のみ、ネイティブアプリを再検討する。

これにより、1つのトラッカーコードベースを維持し、ネイティブ作業を延期する。コンパニオンはマルチカメラフュージョン (#183) のための最初の現実的な「セカンドカメラ」となる。プロトタイプ/手動基準が通過する前に、リアルiPhoneの証拠は #228 からリンクされなければならない。この研究文書によってランタイムの完了は主張されていない。
