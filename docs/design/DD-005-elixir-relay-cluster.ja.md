# DD-005: Elixir クラスタリレー

ステータス: ハーネス実装済み。バックログ: KGM-032、KGM-034に関連。

## 問題

relay-rsは1つのノードをうまく処理します。人気のストリーマーは、地域を超えて数千人の視聴者にファンアウトする必要があり、ノードの障害がストリームを終了させてはならず、ルームはノード間で見つけられる必要があります。これは分散の問題であり、BEAMの得意分野です。

## 目標

- 水平ファンアウト: パブリッシャーは最寄りのエッジノードに接続し、フレームはすべてのノードのサブスクライバーに届く。
- ノードの損失は、そのノードのローカルサブスクライバーのみを削除する。
- KGMフレームはエンドツーエンドで不透明なバイナリのまま。

## トポロジー

```
 tracker --wt--> edge node A ----+
                                  |  Phoenix.PubSub (pg) over dist-erlang
 viewer  <--wt-- edge node B <---+
 viewer  <--ws-- edge node C <---+
```

- 各ノードごとに1つのGenServer（「ルームエージェント」）がPubSubトピック`room:<id>`にサブスクライブし、ローカルサブスクライバーセットを所有します。
- パブリッシャーフレーム: エッジがデータグラムを受信 -> `PubSub.broadcast`でバイナリを送信。同じノード内でのローカル配信はPubSubをバイパスします。
- バックプレッシャー: 各サブスクライバープロセスは1スロットのメールボックスを保持（最新のフレームが勝つ）、KGM-034のセマンティクスを反映します。BEAMのプロセスごとのメールボックスにより、これは簡単です: 送信時に、送信されていない前のフレームを削除します。

## WebTransportの終了

BEAMには成熟したWebTransportサーバーがありません。2つのオプションがあります:

1. Rustサイドカー（推奨）: relay-rsはwt/QUICを終了し、ローカルBEAMノードに対してトリビアルな長さプレフィックス付きTCPまたはUnixソケットプロトコルを話すモードを獲得します。明確な所有権: Rustがパケットを処理し、Elixirが分散を処理します。
2. Rustler NIF埋め込みwtransport: より密接ですが、QUICイベントループがBEAMスケジューラ内で動作するためには、ダーティスケジューラと慎重なライフタイム管理が必要です。

WebSocketサブスクライバーは、同じノードのPhoenix（Bandit）でネイティブに終了し、Safariもカバーします。

## サイズ目標

1パブリッシャー -> 3ノードで5,000サブスクライバー、p99追加リレー遅延< 30 ms、60 fpsで76バイトのフレーム（合計約23 Mbit/sのエグレス; trivial bandwidth、作業はシステムコールとスケジューリングです）。

## 障害モデル

- エッジノードがダウン: そのクライアントはDNS/anycastを介して別のノードに再接続します; ルームエージェントはノードごとに存在するため、グローバルな状態は失われません。
- PubSubパーティション: ルームは各パーティション内で機能し続けます; このデータクラスに対して分割脳の動作を許容可能として文書化します。

## マイルストーン

1. Phoenixアプリ: ws専用のクラスタリレー + 負荷テストハーネス。
2. Rustサイドカープロトコル + wtエッジを1ノードで。
3. マルチリージョンデプロイレシピ（fly.ioまたはプレーンVM） + カオステスト。

## ハーネス証拠

`services/erlang-router/load-test.mjs`は、3ノード、5,000サブスクライバー、1パブリッシャー、最新のみのローカルサブスクライバーメールボックス、およびノード損失の隔離を持つルームエージェントのトポロジーをモデル化します。これは`pnpm test`から実行され、p99ファンアウト遅延が>= 30 msの場合や、失敗したノードが非ローカルサブスクライバーに影響を与える場合に失敗します。

最新のローカル結果: p99 1.07 ms、ノード損失は失敗したノードのサブスクライバーに制限されます。
