# DD-008: キャリブレーションとリターゲティング

ステータス: デザイン。バックログ: KGM-013, KGM-014, KGM-044。

## 問題

ウェブカメラと魅力的なアバターの間には、2つの独立したギャップがあります：

1. 人間 -> 標準チャネル: 顔が異なる。生のブレンドシェイプ範囲は、個人、カメラ、照明によって異なります。（キャリブレーション）
2. 標準チャネル -> アバターコントロール: アバターが異なる。完璧な同期VRMは52チャネルを1:1で必要とし、5表情のVRoidモデルはキュレーションされたマッピングを必要とし、Inochi2Dのパペットは名前付きパラメータを必要とします。（リターゲティング）

Minamoはこれらを分離します：キャリブレーションはトラッカー側に存在し、すでに正規化された状態でワイヤを通過します；リターゲティングはアバターごとにレンダラー側に存在します。

## キャリブレーション（トラッカー側）

ガイドフロー、約30秒：

1. ニュートラルホールド（3秒）：チャネルごとのオフセット o_i = サンプルの中央値。
2. プロンプトされた最大値：大きな口を開ける、広い笑顔、眉を上げる、強い瞬き、遠く左/右/上/下を見る。チャネルごとのゲイン g_i = 1 / p95(最大位相)。
3. 適用されたプレフィルター：w_i' = clamp01((w_i - o_i) * g_i)。
4. サニティクランプ：ゲインは[0.5, 4]の範囲；観測されていないチャネルはアイデンティティを保持。

プロファイルJSON: { name, camera fingerprint, offsets[52], gains[52], deadzones[52], created }。複数のプロファイル；カメラデバイスによって自動選択。

インタラクティブミキサー（KGM-014）は同じ数値を手動で編集するため、データモデルは1つの構造体で2つのUIを持ちます。

## リターゲティング（レンダラー側）

アバターごとのマッピングドキュメント：

```
{ "schema": "minamo.expression-map.v1",
  "name": "Creator rig",
  "targets": [
    { "out": "happy",           // VRM表情またはInochi2Dパラメータ
      "expr": [ ["mouthSmileLeft", 0.5], ["mouthSmileRight", 0.5] ],
      "curve": "easeIn", "clamp": [0, 1] } ] }
```

- 完璧な同期検出：アバターがARKit名の45以上を公開している場合、アイデンティティマッピングを自動的に生成し、エディタをスキップします。
- エディタ：出力を選択し、重み付けされたソースを追加し、カーブプリセットをドラッグし、リアルタイムで確認します。JSONのエクスポート/インポート；アバターごとにコミュニティ共有可能。
- 同じドキュメント形式がVRM表情とInochi2Dパラメータを駆動します（DD-004）；出力の名前空間のみが異なります。
- スキーマ: [../product/expression-mapping.schema.json](../product/expression-mapping.schema.json)。

## なぜワイヤの前に正規化するのか

キャリブレーションされたチャネルを送信することは、すべてのレンダラー、録音（DD-007）、およびコラボレーションの仲間が同じ正規化された信号を視認できることを意味し、キャリブレーションを視聴者に配布する必要がありません。コスト：生の未キャリブレーションデータは下流で回復できないため、これは許容されます；録音はメタデータにプロファイルを記録します。
